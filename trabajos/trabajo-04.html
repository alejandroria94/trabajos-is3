Temas centrales:

Spring Boot Dockerfile Build vs Runtime Tamaño de imagen Seguridad Despliegue Multi-stage Volúmenes Redes

1. Objetivo

Construir y analizar distintos enfoques de contenerización para una aplicación Spring Boot, comparando su impacto sobre el tamaño de la imagen, el tiempo de construcción, el despliegue, la seguridad y la organización de servicios. Al finalizar, el estudiante deberá justificar por qué conviene separar la compilación de la ejecución, cuándo usar multi-stage, y por qué los volúmenes y las redes son fundamentales en escenarios reales.

2. Objetivos de aprendizaje

  • Comprender el papel del Dockerfile, la separación entre build y runtime, y el enfoque multi-stage en Spring Boot.
  • Analizar el impacto de las decisiones de construcción sobre el tamaño de la imagen, la seguridad y el despliegue.
  • Aplicar buenas prácticas de construcción y optimización de imágenes para aplicaciones Spring Boot.
  • Identificar la importancia de los volúmenes para la persistencia de datos.
  • Comprender el uso de redes para la comunicación entre contenedores.
  • Argumentar con claridad por qué estas prácticas son relevantes en producción.

3. Requisitos previos

  • Docker Desktop o Docker Engine instalado.
  • Java 21 instalado.
  • Maven instalado o wrapper mvnw disponible.
  • Editor de texto o IDE.
  • Terminal o consola.
Importante: este laboratorio puede resolverse usando una aplicación Spring Boot existente o creando una mínima desde cero.

6. Preparación del proyecto Spring Boot

6.1 Opción A: usar un proyecto Spring Boot existente

Si ya tiene un proyecto Spring Boot funcional, puede usarlo para el laboratorio. Debe cumplir al menos con lo siguiente:

  • Debe compilar con Maven.
  • Debe generar un archivo .jar.
  • Debe poder ejecutarse localmente.
  • Debe exponer un puerto HTTP, normalmente el 8080.

6.2 Opción B: crear un proyecto Spring Boot mínimo

Si no tiene un proyecto previo, cree una aplicación mínima con Spring Boot.

6.3 Estructura mínima esperada

La estructura básica del proyecto debe verse aproximadamente así:

laboratorio-springboot/ ├── src/ │ ├── main/ │ │ ├── java/ │ │ │ └── com/ejemplo/demo/ │ │ │ ├── DemoApplication.java │ │ │ └── HolaController.java │ │ └── resources/ │ │ └── application.properties ├── pom.xml └── mvnw (opcional)

6.4 Clase principal

Archivo: DemoApplication.java

package com.ejemplo.demo; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure.SpringBootApplication; @SpringBootApplication public class DemoApplication { public static void main(String[] args) { SpringApplication.run(DemoApplication.class, args); } }

6.5 Controlador sencillo

Archivo: HolaController.java

package com.ejemplo.demo; import org.springframework.web.bind.annotation.GetMapping; import org.springframework.web.bind.annotation.RestController; @RestController public class HolaController { @GetMapping("/hola") public String hola() { return "Hola desde Spring Boot con Docker 🚀"; } }

6.6 Archivo de propiedades

Archivo: src/main/resources/application.properties

spring.application.name=demo server.port=8080

6.7 Dependencias mínimas del pom.xml

El proyecto debe incluir al menos Spring Boot y Web.

<dependencies> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> </dependencies>
Si el estudiante ya tiene un proyecto más completo, no necesita crear uno nuevo. Puede adaptar el laboratorio al proyecto existente, pero debe conservar la lógica de comparación entre enfoques.

8. Primer enfoque: compilar dentro del Dockerfile

8.1 Crear Dockerfile incorrecto

Cree un archivo llamado Dockerfile.bad.

FROM maven:3.9-eclipse-temurin-21 WORKDIR /app COPY . . RUN mvn clean package CMD ["java", "-jar", "target/demo-0.0.1-SNAPSHOT.jar"]

8.2 Construir imagen

docker build -f Dockerfile.bad -t spring-docker-bad .

8.3 Ejecutar contenedor

docker run --rm -p 8080:8080 spring-docker-bad

8.4 Probar el endpoint

9. Segundo enfoque: compilación separada y Docker solo ejecuta

9.1 Crear Dockerfile optimizado

Cree un archivo llamado Dockerfile.good.

9.2 Construir imagen

docker build -f Dockerfile.good -t spring-docker-good .

9.3 Ejecutar contenedor

docker run --rm -p 8080:8080 spring-docker-good

9.4 Probar endpoint

10. Tercer enfoque: multi-stage con Spring Boot

10.1 Crear Dockerfile multi-stage

Cree un archivo llamado Dockerfile.multi.

# Etapa 1: build FROM maven:3.9-eclipse-temurin-21 AS build WORKDIR /app COPY . . RUN mvn clean package -DskipTests # Etapa 2: runtime FROM eclipse-temurin:21-jdk-alpine WORKDIR /app COPY --from=build /app/target/demo-0.0.1-SNAPSHOT.jar app.jar ENTRYPOINT ["java", "-jar", "app.jar"]

10.2 Construir imagen multi-stage

docker build -f Dockerfile.multi -t spring-docker-multi .

10.3 Ejecutar contenedor

docker run --rm -p 8080:8080 spring-docker-multi

10.4 Verificar endpoint

http://localhost:8080/hola

12. Persistencia con volúmenes

12.1 Simular persistencia de datos

Para observar el comportamiento de los volúmenes, cree un volumen y escriba un archivo desde un contenedor.

docker volume create datos-spring-lab docker run --rm -v datos-spring-lab:/datos alpine sh -c "echo 'Persistencia Spring Boot' > /datos/info.txt"

12.2 Verificar el contenido del volumen

docker run --rm -v datos-spring-lab:/datos alpine cat /datos/info.txt
El archivo sigue existiendo aunque el contenedor anterior ya no esté. Eso demuestra persistencia fuera del ciclo de vida del contenedor.

13. Comunicación entre contenedores con redes

13.1 Crear una red Docker

docker network create red-spring-lab

13.2 Levantar un contenedor de base de datos

docker run -d --name db-lab --network red-spring-lab -e MYSQL_ROOT_PASSWORD=123456 -e MYSQL_DATABASE=demo mysql:8.0

13.3 Probar conectividad desde otro contenedor

Para no modificar aún la aplicación Spring Boot, haga una prueba conceptual de conectividad desde otro contenedor:

docker run --rm --network red-spring-lab alpine sh -c "ping -c 2 db-lab"
Ambos contenedores pueden encontrarse por nombre porque pertenecen a la misma red Docker.
