1. Objetivo

Analizar cómo las decisiones de construcción de una imagen Docker impactan el tamaño, la seguridad, la eficiencia y el despliegue en entornos productivos. Además, comprender el valor práctico del enfoque multi-stage, así como la importancia de los volúmenes para la persistencia de datos y de las redes para la comunicación entre contenedores.

2. Objetivos de aprendizaje

  • Comprender conceptualmente el papel del Dockerfile, la separación entre build y runtime, y el enfoque multi-stage.
  • Analizar técnicamente el impacto que tienen las decisiones de construcción sobre el tamaño de la imagen, la seguridad y el despliegue.
  • Aplicar buenas prácticas relacionadas con la optimización de imágenes, la separación de responsabilidades y el uso adecuado de multi-stage.
  • Identificar la importancia de los volúmenes para la persistencia de datos y de las redes para la comunicación entre contenedores.
  • Argumentar con claridad y de manera estructurada por qué estas prácticas son relevantes en escenarios reales de desarrollo y producción.

3. Contexto del taller

En muchos proyectos, especialmente al iniciar el uso de Docker, es común colocar todo el proceso dentro del mismo Dockerfile: instalar dependencias, compilar, empaquetar y ejecutar. Aunque esto puede parecer práctico, en realidad genera imágenes más pesadas, menos seguras y poco adecuadas para producción.

Adicionalmente, cuando se usan contenedores con bases de datos o múltiples servicios, aparecen otros temas fundamentales: la persistencia de información mediante volúmenes y la comunicación entre servicios mediante redes. Por ello, este taller amplía el análisis hacia estos dos conceptos.

Este trabajo se centra en responder y reforzar los siguientes interrogantes:

  • ¿Por qué no compilar en Dockerfile?
  • ¿Qué sucede con el tamaño de una imagen si incluimos la compilación dentro del Dockerfile?
  • ¿Qué ventajas tiene separar la compilación del proceso de ejecución cuando trabajamos con contenedores?
  • ¿Cómo impacta esta separación en la seguridad y en el despliegue en entornos productivos?
  • ¿Por qué los volúmenes son fundamentales cuando un contenedor maneja datos?
  • ¿Por qué las redes son necesarias cuando varios contenedores deben comunicarse entre sí?

5. ¿Dónde mirar la información solicitada?

5.1 Tamaño de la imagen

Para revisar el tamaño de una imagen construida, ejecuta:

docker images

Observa especialmente la columna SIZE. Allí podrás comparar si una imagen ocupa más espacio que otra y relacionarlo con lo que contiene internamente.

5.2 Tiempo de build

Para analizar el tiempo de construcción, ejecuta:

docker build -t app-test .

Observa cuánto tarda el proceso. Si deseas, puedes registrar el tiempo manualmente o tomar captura del resultado final del build.

5.3 Tiempo de despliegue o arranque

Para analizar el tiempo de ejecución o arranque del contenedor, ejecuta:

docker run -p 8080:8080 app-test

Observa cuánto tarda en iniciar la aplicación y en quedar disponible.

5.4 Seguridad

La seguridad en este taller se analiza de manera conceptual. Revisa si la imagen final incluye elementos que no deberían estar en producción, por ejemplo:

  • Maven o herramientas de construcción
  • Archivos fuente innecesarios
  • Dependencias temporales de compilación
  • Artefactos intermedios que no se requieren para ejecutar

5.5 Persistencia con volúmenes

Para analizar el uso de volúmenes, observa si los datos del contenedor sobreviven aun cuando este se elimina. Puedes apoyarte en comandos como:

docker volume ls docker inspect nombre-contenedor docker run -v datos:/ruta/interna imagen

Lo importante es identificar si la información queda solo dentro del contenedor o si se almacena por fuera de él.

5.6 Comunicación mediante redes

Para analizar redes, revisa cómo se comunican varios contenedores entre sí. Puedes apoyarte en comandos como:

docker network ls docker inspect nombre-red docker run --network nombre-red ...

Debes identificar si los contenedores se encuentran aislados o si pueden interactuar entre ellos por nombre o por servicio.

5.7 Despliegue en producción

En tus respuestas debes considerar cómo el tamaño, la seguridad, la persistencia y la conectividad de la imagen pueden afectar:

  • La velocidad de descarga de la imagen
  • El consumo de almacenamiento
  • El tiempo de despliegue en servidores
  • La facilidad de mantenimiento
  • La confiabilidad del artefacto que llega a producción
  • La persistencia de información importante
  • La comunicación correcta entre servicios

6. Paso a paso del taller

Paso 1. Comprensión del problema

Lee el siguiente Dockerfile y observa que en él se realiza tanto la compilación como la ejecución de la aplicación dentro de la misma imagen.

FROM maven:3.9-eclipse-temurin-21 WORKDIR /app COPY . . RUN mvn clean package CMD ["java", "-jar", "target/app.jar"]

Actividad 1. Responde:

  1. ¿Qué herramientas quedan incluidas en la imagen final?
  2. ¿Por qué esta imagen puede resultar más pesada de lo necesario?
  3. ¿Qué archivos adicionales podrían quedar dentro de la imagen?
  4. ¿Consideras este Dockerfile adecuado para producción? Justifica.

Paso 2. Análisis del tamaño de imagen

Si tienes Docker disponible, construye la imagen y revisa su tamaño.

docker build -t dockerfile-incorrecto . docker images

Actividad 2. Analiza y responde:

  1. ¿Qué sucede con el tamaño de la imagen si la compilación se realiza dentro del Dockerfile?
  2. ¿Qué consecuencias puede tener una imagen muy grande en descarga, almacenamiento y despliegue?
  3. Si realizaste la prueba, registra el tamaño observado o anexa una captura.

Paso 3. Separar compilación y ejecución

Ahora observa una alternativa en la que la aplicación se compila previamente y Docker se usa únicamente para ejecutar el artefacto generado.

FROM eclipse-temurin:21-jdk-alpine WORKDIR /app COPY app.jar app.jar ENTRYPOINT ["java", "-jar", "app.jar"]

Actividad 3. Analiza este enfoque y responde:

  1. ¿Qué ventajas tiene separar la compilación del proceso de ejecución?
  2. ¿Qué cambia respecto al tamaño de la imagen final?
  3. ¿Qué cambia respecto al tiempo de despliegue?
  4. ¿Qué cambia respecto al control del artefacto liberado?

Paso 4. Comparación práctica entre enfoques

Si tienes acceso a Docker, construye también una imagen basada en el enfoque optimizado y compara resultados.

docker build -t dockerfile-optimizado . docker images

Actividad 4. Compara ambas imágenes:

  1. ¿Cuál es más grande?
  2. ¿Cuál es más adecuada para despliegue en producción?
  3. ¿Cuál descargaría más rápido en un servidor o pipeline?
  4. ¿Qué evidencias encontraste al revisar el tamaño o el proceso de build?

Paso 5. Seguridad y despliegue en producción

En un entorno productivo, una imagen debería contener únicamente lo necesario para ejecutar la aplicación. Incluir herramientas como Maven o archivos de compilación amplía la superficie de ataque y complica el mantenimiento.

Actividad 5. Responde de forma argumentada:

  2. ¿Cómo impacta esta separación en el despliegue en entornos productivos?
  3. ¿Por qué una imagen más pequeña suele ser más conveniente para producción?

Paso 7. Introducción a Multi-Stage

Una alternativa más completa consiste en usar un Dockerfile multi-stage. Este enfoque permite compilar la aplicación en una primera etapa y copiar únicamente el artefacto necesario a una segunda etapa, destinada a la ejecución.

# Etapa 1: build FROM maven:3.9-eclipse-temurin-21 AS build WORKDIR /app COPY . . RUN mvn clean package # Etapa 2: runtime FROM eclipse-temurin:21-jdk-alpine WORKDIR /app COPY --from=build /app/target/app.jar app.jar ENTRYPOINT ["java", "-jar", "app.jar"]

Actividad 6. Explica:

  1. ¿Qué ocurre en la etapa build?
  2. ¿Qué ocurre en la etapa runtime?
  3. ¿Qué elementos pasan de una etapa a otra?
  4. ¿Qué elementos no quedan en la imagen final?
  5. ¿Qué ventajas ofrece multi-stage frente al Dockerfile que compila y ejecuta todo en la misma etapa?

Paso 9. Persistencia con volúmenes

Cuando un contenedor maneja datos, como ocurre con una base de datos, un archivo de configuración o un repositorio de información, es necesario decidir dónde se almacenará esa información. Si todo queda dentro del contenedor, al eliminarlo los datos podrían perderse.

docker run -v datos:/var/lib/mysql mysql:8.0

Actividad 8. Responde:

  1. ¿Qué problema resuelven los volúmenes en Docker?
  2. ¿Qué podría pasar si una base de datos corre en un contenedor sin volumen?
  3. ¿Por qué se dice que los volúmenes separan los datos del ciclo de vida del contenedor?
  4. ¿En qué escenarios reales considerarías obligatorio usar volúmenes?

Paso 10. Comunicación mediante redes

En aplicaciones con varios servicios, como una API y una base de datos, los contenedores deben poder comunicarse. Para ello Docker permite crear redes que facilitan la interacción entre ellos.

docker network create red-app docker run -d --name db --network red-app mysql:8.0 docker run -d --name app --network red-app mi-app

Actividad 9. Responde:

  1. ¿Para qué sirven las redes en Docker?
  2. ¿Por qué no basta con que los contenedores estén simplemente ejecutándose?
  3. ¿Qué ventajas aporta una red Docker cuando una aplicación necesita conectarse a una base de datos?
  4. ¿Qué riesgos o limitaciones podrían surgir si no se configura correctamente la comunicación entre contenedores?

Cierre del taller:

En Docker no solo importa que la aplicación funcione, sino cómo se construye la imagen, cómo se gestionan los datos y cómo se conectan los servicios, ya que todo ello impacta directamente el tamaño, la seguridad, la persistencia, la eficiencia y el comportamiento del despliegue en producción.

Nota: Este taller complementa el trabajo previo sobre Docker y busca afianzar la comprensión de la optimización de imágenes, la separación entre compilación y ejecución, el uso de multi-stage, y la importancia de volúmenes y redes como prácticas fundamentales en escenarios reales.
